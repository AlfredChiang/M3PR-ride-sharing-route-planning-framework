# Reproduction Guide

This document explains how the code maps to the manuscript and how to extend it into a full experimental reproduction.

## 1. Core objective

The paper defines platform profit and transforms the maximization objective into the minimization objective:

```text
P_sub = beta_o * sum(distance of unserved orders)
        + alpha_v * sum(route distance of all vehicles)
```

In the code, this is computed in `OnlinePlanner.run()` after all orders are processed.

The immediate profit-aware decision is implemented through the bound:

```text
tau_q = beta_o * d(origin_q, destination_q) / alpha_v
```

Any insertion whose increased route distance is greater than `tau_q` is dominated by rejecting the order and is excluded in `linear_profit_bounded_insert()`.

## 2. Route schedule representation

The paper writes a vehicle route schedule as:

```text
S_i = <l_0, l_1, ..., l_n>
```

where `l_0` is the current vehicle location.  In this repository:

```python
Vehicle.current_node  # l_0
Vehicle.route         # [l_1, ..., l_n], each as a Stop
```

A new order is inserted by choosing a pair `<l_i, l_j>`:

- `i < j`: insert pickup after `l_i`, insert dropoff after `l_j`;
- `i = j`: insert pickup and dropoff consecutively after `l_i`.

## 3. Implemented algorithms

### Algorithm 1: solution framework

Implemented by `OnlinePlanner.run()`.

Workflow:

1. orders are processed in release-time order;
2. vehicle states are advanced to the order release time;
3. candidate vehicles are generated by dual-side pruning or by using all vehicles;
4. an insertion method is called for each candidate vehicle;
5. the vehicle with minimum increased distance is selected;
6. if no profitable feasible insertion exists, the order is rejected.

### Algorithm 2: dual-side vehicle pruning

Implemented by `dual_side_vehicle_pruning()`.

The paper maintains three per-grid lists:

- vehicle list ordered by expected arrival time;
- temporal grid list ordered by travel time;
- spatial grid list ordered by travel distance.

The current executable scaffold evaluates the same reachability inequalities directly on current vehicle anchors:

```text
time(vehicle, origin) <= max_pickup_delay

time(vehicle, destination) <= max_pickup_delay
                              + time(origin, destination) * (1 + detour_ratio)
```

This is intentionally simpler than the full production index, but it keeps the same pruning spirit and can be replaced by a per-grid indexed implementation later.

### Algorithm 3: cubic insertion

Implemented by `cubic_insert()`.

It explicitly enumerates every insertion pair, builds the new route, checks feasibility by simulating the full route, and computes increased route distance from scratch.

### Algorithm 4: DP-based quadratic insertion

Implemented by `quadratic_dp_insert()`.

It still enumerates insertion pairs, but precomputes:

- arrival time `partial(l_k)`;
- slack time `T(l_k)`;
- onboard load `rho(l_k)`;
- O(1) detour values.

### Algorithm 5: profit-bounded DP-based linear insertion

Implemented by `linear_profit_bounded_insert()`.  After studying the uploaded GreedyDP implementation, this function was updated to use a `RouteDPState` structure corresponding to the reference code's `reach`, `picked`, and `slack` arrays.

Main states:

```text
zeta(l_j): minimum pickup-side detour before delivery position l_j
eta(l_j):  pickup position corresponding to zeta(l_j)
T(l_k):    downstream deadline slack
rho(l_k):  onboard load after l_k
partial(l_k): arrival time at l_k
```

The updated implementation uses time detours for deadline and compatibility checks, and distance detours for the M3PR objective and the profit bound.  This fixes the common reproduction mistake of comparing distance increments directly with time slack.

For each `j`, the code checks:

1. the special case `i = j`;
2. the general case `i < j` using the maintained `zeta/eta` pair;
3. the profit bound `tau_q`.

## 4. How to plug in real datasets

The paper's experiments require four preprocessing steps:

1. obtain road network from OSM/OSMnx or QGIS;
2. partition the study area into 650m x 650m grids;
3. map each trip origin/destination to a grid anchor node;
4. precompute shortest-path distance/time between all grid anchors.

After this, export:

```text
orders.csv
vehicles.csv
matrix.npz
```

Then create a small runner similar to:

```python
from rideshare_m3pr.io import load_matrix_npz, load_orders_csv, load_vehicles_csv
from rideshare_m3pr.planner import OnlinePlanner, PlannerConfig

matrix = load_matrix_npz("matrix.npz")
orders = load_orders_csv("orders.csv")
vehicles = load_vehicles_csv("vehicles.csv", default_capacity=4)

planner = OnlinePlanner(matrix, vehicles, PlannerConfig(algorithm="linear"))
stats = planner.run(orders)
print(stats)
```

## 5. Important caveats

1. The synthetic demo is for code sanity checks, not for reproducing the paper's exact tables or figures.
2. Exact results require the same trip cleaning rules, grid anchors, speed model, vehicle initialization, and random seed as the paper.
3. The current `advance_vehicle_to()` keeps a vehicle at its last visited anchor if it is mid-edge at an order release time.  For exact simulation, replace it with a continuous movement or map-matched interpolation module.
4. The current pruning module is compact.  For large real datasets, implement the full per-grid vehicle list plus temporal/spatial grid lists described in the paper.

## 7. Notes from the uploaded GreedyDP code

The uploaded reference implementation contains a highly useful linear insertion routine in `util.cpp`.  Its `try_insertion()` function first evaluates the consecutive pickup/dropoff case and then maintains a DP array `Det` to record the best pickup-side detour before each delivery position.  In our paper notation, `Det` corresponds to `zeta(l_j)` and its stored index corresponds to `eta(l_j)`.

The reproduction now includes `docs/GREEDYDP_REFERENCE_NOTES.md`, which explains this mapping in detail.

## 8. GreedyDP-style data adapter

The following helpers parse the uploaded reference-style input files:

```python
from rideshare_m3pr.io import load_gdp_taxi_file, load_gdp_order_file
```

Example runner:

```bash
python scripts/run_gdp_inputs.py --matrix matrix.npz --taxi taxi.txt --orders order.txt --algorithm linear
```

A compatible `matrix.npz` is still required because the uploaded package does not include the Chengdu road network and hub-label files.
